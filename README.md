# Highload Auth Service

Микросервис аутентификации и управления аккаунтами, спроектированный для высоких нагрузок.

## Tech Stack
- **Language:** Go (1.25+)
- **Framework:** Fiber v2
- **Database:** PostgreSQL (pgx pool)
- **Cache/Session:** Dragonfly (Redis-compatible)
- **Load Balancer:** Nginx
- **Containerization:** Docker & Docker Compose

## Architectural Decisions

### 1. Token Strategy (JWT + Opaque)
*   **Access Token:** Stateless JWT. Позволяет проверять авторизацию без запроса в БД/Кэш, что критично для Highload.
*   **Refresh Token:** Opaque string (случайные байты) в Dragonfly. Позволяет реализовать мгновенный отзыв сессии (logout) и Token Rotation, не раздувая БД.

### 2. Horizontal Scaling
*   Приложение контейнеризировано и подготовлено к запуску нескольких реплик.
*   Использование Nginx в качестве Reverse Proxy позволяет распределять нагрузку между инстансами, снижая задержки (p95) при интенсивных операциях (например, хэшировании паролей).

### 3. Session Storage (Dragonfly)
*   Выбран Dragonfly вместо классического Redis, так как он показывает лучшую производительность на многоядерных системах, что соответствует целям Highload-проекта.

### 4. Crypto Worker Pool (Изоляция CPU-нагрузки)
*   Тяжелые операции хэширования (`bcrypt`) вынесены в отдельный пул воркеров.
*   Это предотвращает блокировку основных потоков обработки HTTP-запросов и гарантирует стабильную работу "легких" ручек (например, получение профиля) даже при массовых попытках входа.

### 5. Database Optimization
*   Использование `pgxpool` для эффективного управления соединениями с PostgreSQL.
*   Слоистая архитектура (Handler -> Usecase -> Repository) для упрощения тестирования и масштабирования.

## How to Run

1.  Настройте `.env` (см. `.env.example`).
2.  Запустите инфраструктуру и приложение:
    ```bash
    docker-compose up --build -d
    ```
3.  Сервис будет доступен по адресу: `http://localhost:8080/api/v1`

## Load Testing
Для проверки производительности используется k6:
```bash
k6 run scripts/load_test.js
```

## Performance Insights & Conclusions (по итогам тестов)

В ходе нагрузочного тестирования (300 VUs, k6) были сделаны следующие выводы:

1.  **Критичность ресурсов CPU:** Операции аутентификации (bcrypt/argon2) крайне требовательны к процессору. Ограничение контейнера до 0.5 CPU привело к деградации задержек (p95) до **~9 сек**. После снятия лимитов задержка упала до **~2.6 сек**.
2.  **Эффективность репликации:** Переход от одного инстанса к кластеру из 3 реплик за Nginx позволил увеличить пропускную способность (RPS) на **~12%** и снизить p95 на **~25%** на том же железе за счет лучшей утилизации ядер.
3.  **Стабильность под давлением:** Система сохранила **0.00% ошибок** при 300 одновременных пользователях, выполняющих полный цикл действий без пауз. Это подтверждает надежность связки Dragonfly + pgxpool.
4.  **Главный Bottleneck:** Основным ограничивающим фактором является хэширование паролей. Для дальнейшего роста RPS требуется либо наращивание вычислительных мощностей (вертикальное масштабирование), либо вынос хэширования в отдельные микросервисы (Workers).
5.  **Эффективность Worker Pool:** Внедрение пула воркеров позволило снизить **медианную задержку на 25%** и в 20 раз ускорить обработку легких запросов (минимальная задержка упала с 400мкс до 20мкс) за счет изоляции блокирующих CPU-задач.
